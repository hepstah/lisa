# Quick Task 260413-l32: Pre-Hardware Enhancements Summary

Pre-hardware software hardening: backend test suite no longer depends on live Anthropic credits, and every LLM intent call in dev mode leaves a structured debug trail in the command_log row plus the dashboard expanded command row.

## What was built

### Part A -- Test isolation from Anthropic (Task 1)

`backend/tests/conftest.py` gains two additions on top of the existing fixtures:

1. `_fail_on_live_anthropic_key` (session-scoped, autouse). Uses `pytest.MonkeyPatch()` to wrap `anthropic.AsyncAnthropic.__init__`; any constructor call with `api_key` starting with `sk-ant-` raises `RuntimeError("Live Anthropic key detected in test run. Use the mock_llm_intent fixture or a dummy key.")`. Dummy keys like `test-key-123` pass through unchanged.
2. `mock_llm_intent` (function-scoped). Monkeypatches `LLMIntentService.parse_intent` with an async helper that routes responses by input text. Exposes `.set_response(text, intent=None, raise_=None)`. Default fallback returns `IntentResult(intent=None, debug={...})`. `IntentResult` is imported lazily inside the fixture so conftest stays collectible independent of Task 2 ordering.

Additionally, conftest now forces `LISA_ANTHROPIC_API_KEY` and `LISA_OPENAI_API_KEY` to `test-key-123` at import time. Without this, the `.env`-loaded live key in this developer env would instantiate `AsyncAnthropic(api_key="sk-ant-...")` during FastAPI lifespan and trip the safety net before any test ran. Dummy env vars satisfy the safety net and the existing settings validation, and production behavior is unchanged (real deployments still load real keys from `.env`).

Behavior of the safety net:

- Dummy key prefix (anything not `sk-ant-`): passes through to original `__init__`.
- `sk-ant-` prefix: raises `RuntimeError` with a clear message pointing at the mock fixture.
- Session cleanup via `mp.undo()` on yield unwind so subsequent pytest sessions are unaffected.

### Part B -- Dev-mode LLM debug capture (Tasks 2 + 3)

**Backend:**

- `llm_intent_service.py`: new `IntentResult(intent, debug)` dataclass. `parse_intent` now returns `IntentResult`; it always populates a debug dict with `input_text`, `devices_seen`, `decision`, `usage` (from `response.usage`), and `stop_reason`. Decision variants:
  - Tool-use: `{tool_used: True, device_id, action, confirmation}`
  - Plain text: `{tool_used: False, text}`
  - (Error variant is built by voice_pipeline, since parse_intent re-raises on APITimeout / APIConnection / APIError exactly as before.)
- `voice_pipeline.py`: imports `settings` from `lisa.config` and adds a `_dump_debug` helper. Every LLM path (success, unknown intent, LLMTimeoutError, LLMError) builds a debug dict and JSON-encodes it only when `settings.dev_mode` is true. On success, the JSON string is threaded into `DeviceService.execute_command` via the new `llm_debug` kwarg. On pipeline errors, the JSON string is included in the returned dict so `api/commands.text_command` can persist it via its pipeline-error INSERT.
- `device_service.py`: `execute_command` accepts `llm_debug: Optional[str]` and forwards it to `_log_command` on all four call sites (rejected, success, ConnectionError, generic Exception). `_log_command` writes it to the new `llm_debug` column as an opaque passthrough -- DeviceService never inspects the value.
- `db.py`: `llm_debug TEXT` appended to the `command_log` CREATE TABLE. An idempotent `ALTER TABLE command_log ADD COLUMN llm_debug TEXT` runs after the `executescript` block inside a try/except on `aiosqlite.OperationalError` so existing dev DBs migrate silently.
- `models.py`: `CommandRecord.llm_debug: Optional[dict] = None` appended.
- `api/commands.py`:
  - `get_command_history` reads `r["llm_debug"]` and defensively `json.loads` it; `ValueError` / `TypeError` -> `None`. Passes the parsed dict into `CommandRecord(...)`.
  - `text_command` pipeline-error INSERT adds the `llm_debug` column + bind param (value = `result.get("llm_debug")`, which is already a JSON string from voice_pipeline or None in prod mode).
  - Before broadcast, `text_command` parses `result["llm_debug"]` from a JSON string back to a dict (inside try/except) so the WebSocket `command_logged` payload carries the same shape as the history endpoint. Without this, the frontend would see a JSON string on live updates and a dict on history fetch.
  - `_log_unknown` and `_log_no_match` are byte-for-byte unchanged (verified against the pre-task commit). These are the Phase-1 fallback parser path and do not involve the LLM.

**Frontend:**

- `dashboard/src/api/types.ts`: `LlmDebug` interface with a discriminated union on `decision` (`tool_used: true`, `tool_used: false`, or `{error}`). `CommandRecord.llm_debug?: LlmDebug | null`.
- `dashboard/src/components/CommandHistory.tsx`: new `LlmDebugSection` component rendered inside the existing expanded `TableRow` only when `cmd.llm_debug` is truthy. Contains:
  - Header `LLM Debug:` styled like the other expanded-row labels.
  - One-line decision summary branching on the three variants. Tool variant renders `Tool: control_device(device=<id>, action=<action>)`. Text variant renders `No tool -- text: "<first 80 chars>"`. Error variant renders `Error: <error>` in `text-red-500`.
  - Stats row `input <N> tok | output <N> tok | stop: <reason>` -- hidden on the error variant.
  - Collapsible `<details>` titled `Devices seen (<count>)` containing a small three-column table.
- Production dashboards (`llm_debug` null) see the expanded row exactly as before.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocker] Forced dummy API keys in conftest.py**
- Found during: Task 1 verification (first run).
- Issue: The safety net correctly tripped on the `.env`-loaded live Anthropic key during the FastAPI lifespan. `anthropic.AsyncAnthropic(api_key="sk-ant-...")` is constructed inside `lisa.main.lifespan`, which runs before any test body, so the safety net caught it (as designed) and blocked the whole test collection.
- Fix: Added two lines to conftest.py right after the `LISA_DEV_MODE` / `LISA_DB_PATH` env writes:
  ```python
  os.environ["LISA_ANTHROPIC_API_KEY"] = "test-key-123"
  os.environ["LISA_OPENAI_API_KEY"] = "test-key-123"
  ```
  pydantic-settings prefers env vars over `.env`, so this shadows the live key only for the pytest process. Production and normal uvicorn runs are unaffected.
- Files modified: `backend/tests/conftest.py`
- Commit: first Task 1 commit (71a14f0)

### Ambiguities flagged during execution (per output spec)

- **Test assertion relaxation (`test_text_command_unknown_pattern_rejected`, `test_text_command_no_matching_device_rejected`):** these tests previously asserted against Phase-1 fallback parser strings (`"Could not understand"`, `"No device found"`) that originate in `_log_unknown` / `_log_no_match`. On the LLM-mocked path, those helpers are never reached; the unknown-intent path emits `MSG_UNKNOWN_INTENT` (`"I didn't understand that. ..."`). Relaxed assertions per plan guidance: the first test now checks `data["status"] == "rejected"` AND `"didn't understand" in data["error_message"]`; the second checks `data["status"] == "rejected"` only. Flagged here as requested.
- **WS broadcast payload normalization:** added inside `text_command` to parse the JSON string back to a dict before `manager.broadcast(...)`. Without this, the frontend would see a JSON string on live `command_logged` events and a parsed dict on history-fetch events -- mismatching shapes on the same `LlmDebug` contract. The normalization is wrapped in a try/except so malformed strings silently become `None` rather than crashing the broadcast path.
- **conftest forced dummy keys:** documented above.

## Open risk reminders (from the spec)

- Informal schema migration via `try/except ALTER TABLE` works for a single-home SQLite dev DB but would not scale to a managed service. Fold into a migration framework if we ever grow one.
- The `_fail_on_live_anthropic_key` safety net could false-positive if a future integration test needs a real `AsyncAnthropic` client. Opt-out via a pytest marker is easy to add when that need actually materializes.
- `parse_intent` return type changed from `Optional[DeviceIntent]` to `IntentResult`. Backwards-incompatible; the only callers are `voice_pipeline.py` and the tests, both updated here.
