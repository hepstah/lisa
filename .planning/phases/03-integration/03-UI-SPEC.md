## PipelineStatus Component Specification

### Visual Structure

The PipelineStatus component mirrors the StatusBar dot+label pattern (right side of header) but is placed as a standalone indicator in the main content area or integrated into the existing StatusBar.

**Recommended placement:** Inside the existing `StatusBar` header, to the left of the WebSocket connection indicator. This keeps all system status in one row and avoids introducing a new layout region.

```
+---------------------------------------------------------------+
| Lisa        [dot] Listening    [dot] Connected                 |
+---------------------------------------------------------------+
```

### Layout

- Container: `flex items-center gap-2` (8px gap between dot and label)
- Dot: `inline-block h-2 w-2 rounded-full` (same as StatusBar connection dot)
- Label: `text-sm text-muted-foreground` (same as StatusBar connection label)
- Hidden on small screens: `hidden sm:inline` for label text (dot always visible)
- Screen reader: `<span className="sr-only">` with full status text

### State Rendering

| State | Dot Class | Label Text | Animation |
|-------|-----------|------------|-----------|
| listening | `bg-emerald-500` | "Listening" | none (steady state) |
| processing | `bg-amber-500 motion-safe:animate-pulse` | "Processing" | pulse (matches StatusBar connecting pattern) |
| responding | `bg-blue-500` | "Responding" | none |
| error | `bg-red-500` | "Error" | none |
| offline | `bg-zinc-500` | "Offline" | none |

### Behavior

- Default state on mount: "offline" (per RESEARCH.md Open Question 3)
- Transitions: Each WebSocket `pipeline_status` event replaces the current state immediately (no transition animations, no queuing -- per RESEARCH.md Pitfall 3)
- On WebSocket disconnect: Revert to "offline" (handled in useWebSocket onclose, which triggers re-render)

## Interaction Contract

### WebSocket Event Flow

1. Backend VoiceLoop emits `{ type: "pipeline_status", status: "listening" }` when it starts
2. On wake word detection: emits `processing`
3. After STT + LLM completes: emits `responding`
4. After TTS playback: emits `listening`
5. On error: emits `error`, then `listening` after recovery
6. Dashboard displays the most recent status; no buffering or animation between states

### Integration with Existing App.tsx Pattern

The `usePipelineStatus` hook follows the same pattern as `useDevices` and `useCommands`:
- Returns `{ status, handleWsEvent }`
- The `handleWsEvent` callback is added to `handleWsMessage` in App.tsx alongside existing handlers
- The `status` value is passed to the PipelineStatus component (or StatusBar, if integrated there)
